**结论：**

1. **同会话复用（same.sh）的耗时是新建会话（diff.sh）的约 1/5（按平均）**，p50 节省 ~3×、长尾节省 ~12–15×。
2. 新建会话模式下，长尾被严重拉长，且会出现硬性配额失败（HTTP 403）。
3. 在新建会话的场景下，沙箱的生命周期管理是性能与可用性的瓶颈。

## 7. 根因分析

| 现象 | 根因 |
| --- | --- |
| 第 21、22 次请求 HTTP 403 | 沙箱模板硬上限 20 并发；前 20 次会话创建的沙箱未及时释放 |
| 第 6、12、17、20 次请求耗时 5–17 s | 同一原因：创建沙箱排队等待空闲槽位 |
| 第 25 次及之后无 `Finished` 行 | 日志/脚本被截断；测试未跑完 100 次 |
| 请求 23、24 无 `Response:` 块 | 客户端 718ms 即返回，但响应输出未在日志中可见（可能流式输出被截断或写入缓冲未刷新） |

## 8. 建议

1. **重跑前清理沙箱**：在重跑 `diff.sh` 前，先调用平台 API 或控制台释放旧沙箱，确保配额从 0 开始累积。
2. **脚本串行化已生效，但配额仍是瓶颈**：当前脚本本质串行，串行下也会触发 20 上限 → 配额是**模板级并发硬限**，与脚本并发度无关。
3. **会话复用是更优选择**：若业务允许，应使用 `same.sh` 模式（同一 session 复用），耗时和稳定性均显著优于新建会话。
4. **日志截断问题**：建议 `diff.sh` 中给 `agentarts invoke` 加 `2>&1` 重定向并 `tee` 到日志，避免 stdout 缓冲导致响应体丢失；并在末尾捕获 `set -o pipefail` 与退出码，对失败重试。
5. **重跑建议**：在配额已清零的环境下重跑完整 100 次，并将 `diff.sh` 改为：

   ```bash
   agentarts invoke '{"message": "hello,world"}' --session "$SESSION_ID" 2>&1 | tee -a diff.log
   ```
